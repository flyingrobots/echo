Parallel Policy Matrix

Compare shard assignment and delta accumulation strategies across multiple loads. Raw data lives in parallel-policy-matrix.json; this page is the lightweight static view.

Goal, Method, and Metric

This study isolates scheduler topology overhead inside execute_parallel_with_policy. It is not an end-to-end engine trace and it is not measuring semantic conflicts; every load uses the same synthetic independent workload so the comparison stays focused on shard assignment and delta accumulation strategy.

What Each Case Runs

  1. Build a fresh graph store with N independent benchmark nodes.
  2. Create one ExecItem per node using the same touch_executor.
  3. Run that identical item set under one execution policy at one load.
  4. Black-box the returned deltas so the work is not optimized away.

How It Is Measured

  • Harness: Criterion benchmark group parallel_policy_matrix
  • Loads: 100, 1000, 10000 items
  • Warmup: 2s
  • Measurement window: 5s
  • Sample size: 40
  • Primary metric: mean wall-clock time per invocation
  • Confidence interval: Criterion 95% CI from estimates.json

Lower mean runtime is better. The charts and per-load tables rank cases by raw mean runtime. The overall scoreboard normalizes each case against the best runtime at the same load, then averages that slowdown so mixed-load winners are not biased by the largest benchmark size.

What This Compares

  • dynamic_per_worker: dynamic shard stealing, one delta per worker
  • dynamic_per_shard: dynamic shard stealing, one delta per shard
  • static_per_worker: round-robin shard assignment, one delta per worker
  • static_per_shard: round-robin shard assignment, one delta per shard
  • dedicated_per_shard: one worker thread per non-empty shard, one delta per shard
