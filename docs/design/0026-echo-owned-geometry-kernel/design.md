# 0026 — Echo-Owned Geometry, Collision, And Lighting Kernel

_Make geometry an Echo-owned deterministic contract surface: authored schemas
define linear algebra, shapes, ray queries, overlap queries, swept queries, and
contact patches; authored collision and lighting schemas define solver,
constraint, joint, broad-phase, and ray-tracing math; Wesley emits Rust and
TypeScript payload surfaces; `warp-math` and `warp-geom` provide the executable
authority; jedit consumes geometry through Echo instead of maintaining local
ray-tracing primitives._

Legend: `MATH`

Depends on:

- `0024 — Universal LE Binary Codec`
- `crates/warp-math`
- `crates/warp-geom`
- Wesley generated contract hosting
- jedit title-scene ray tracing and title-scene acceleration work

Supersedes / expands:

- The unmerged `design/echo-3d-math` branch, which documents 3D math
  primitives only.

## Why this cycle exists

Echo already has deterministic math and early geometry substrate:

- `crates/warp-math` owns `Vec3`, `Quat`, `Mat4`, deterministic trig,
  scalar policy, fixed-point helpers, and PRNG support.
- `crates/warp-geom` owns `Aabb`, `Transform`, temporal sweep proxies, and
  an AABB-based broad-phase scaffold.
- `crates/echo-scene-port` and `crates/echo-scene-codec` own a small
  rendering-oriented scene port with `Sphere` and `Cube` node shapes.

Those are useful, but they are not yet the Echo geometry platform that jedit
and future applications need.

The missing layer is an authored, generated, cross-language geometry contract:

- GraphQL SDL that names math primitives, shape primitives, query requests,
  query responses, and contact patches.
- Rust types and algorithms backed by `warp-math` / `warp-geom`.
- TypeScript types and codecs generated by Wesley from the same schema.
- Canonical little-endian bytes and golden fixtures for every primitive.
- Deterministic overlap, ray intersection, swept collision, solver,
  constraint, joint, stacking, broad-phase, and lighting APIs.
- Batch query surfaces suitable for title-screen rendering and other
  high-throughput users.

jedit currently carries local title-scene ray tracing and acceleration logic in
TypeScript. That made sense while proving the visual surface, but it is the
wrong long-term boundary. If jedit is an Echo-powered product, then geometry
queries must be Echo geometry queries. jedit may still own scene authorship,
lighting, materials, terminal presentation, and product behavior. Echo should
own deterministic geometry primitives, canonical payloads, broad-phase
acceleration, and narrow-phase query semantics.

## Ownership model

Echo currently hosts the seed geometry semantics. After Bunny is established,
generally reusable graphics primitives move toward Bunny. Echo keeps causal
authority over how those primitives are admitted, retained, queried, and proven.
Applications own domain meaning and presentation.

| Layer        | Owns                                                                  | Does not own           |
| ------------ | --------------------------------------------------------------------- | ---------------------- |
| Bunny        | reusable math, geometry, query, mesh, optics, and schema contracts    | causal runtime facts   |
| `warp-math`  | scalar policy, vectors, matrices, quaternions, deterministic ops      | application entities   |
| `warp-geom`  | shapes, transforms, queries, contact patches, acceleration contracts  | renderer materials     |
| GraphQL SDL  | primitive names, field order, query/result contracts, schema identity | app-specific nouns     |
| Wesley       | Rust/TypeScript types, codecs, fixture metadata                       | geometry algorithms    |
| Echo runtime | admission, canonical bytes, retained evidence, causal query identity  | mutable renderer state |
| jedit        | scene authorship, lighting, material choice, terminal pixels, UX      | geometry authority     |

Authority order:

1. Executable runtime behavior in Bunny, `warp-math`, and `warp-geom`.
2. Cross-language golden fixtures and deterministic command output.
3. Generated Rust / TypeScript code and schema hashes.
4. GraphQL SDL and design docs.
5. jedit adapter code.

Design docs explain the target. They do not prove it.

### Bunny Extraction Update

The long-term invariant is:

```text
Bunny owns reusable graphics primitives.
Echo owns causal use of those primitives.
```

This means Echo's deterministic math and geometry crates are extraction
candidates, not permanent project boundaries. Move a primitive toward Bunny when
it is pure graphics substrate: scalars, vectors, matrices, quaternions,
transforms, shapes, contacts, rays, overlap tests, swept tests, broad phase,
mesh contracts, optics, or lighting math.

Keep a primitive in Echo when it names causal authority: strands, braids,
intents, readings, WSC storage, retained evidence, transaction semantics,
admission policy, or provenance.

## Vocabulary

Use dimensional suffixes in generated contracts where ambiguity matters:

- `Vector2`, `Vector3`, `Vector4`
- `Matrix2x2`, `Matrix3x3`, `Matrix4x4`
- `Aabb2`, `Aabb3`
- `Ray2`, `Ray3`
- `Segment2`, `Segment3`
- `Triangle2`, `Triangle3`
- `HalfSpace2`, `HalfSpace3`
- `Obb3`, `Box2`

Use renderer-neutral names. Echo should not expose names such as
`THREE.Vector3`, `TitleBunny`, `BijouCell`, or `JeditMesh` in geometry core.

## Contract surface

The original Echo-local SDL plan was:

```text
schemas/runtime/echo-math.graphql
schemas/runtime/echo-geometry-2d.graphql
schemas/runtime/echo-geometry-3d.graphql
schemas/runtime/echo-geometry-query.graphql
```

After the Bunny decision, the reusable primitive schemas should instead live in
Bunny-owned schema files such as:

```text
schemas/bunny/v0/graphics.graphql
```

Echo can generate or vendor bindings from Bunny schemas, but Echo should not
become the owner of shared graphics schemas.

Those modules should be importable by application contracts without making
application nouns part of Echo core.

}

    x: EchoF32!

    columns: [EchoF32!]!

- `Matrix2x2.columns.length == 4`

Matrix storage is column-major to align with the existing `warp-math::Mat4`

## Query result contracts

### Contact point

```graphql
type ContactPoint3 {
    point: Vector3!
    normal: Vector3!
    penetrationDepth: EchoF32!
}

type ContactPatch3 {
    normal: Vector3!
    points: [ContactPoint3!]!
}
```

The patch normal points from `shapeA` toward `shapeB`. Each contact point is in
world coordinates. `penetrationDepth` is non-negative and measured along the
patch normal.

For 2D, the same idea uses `Vector2`:

```graphql
type ContactPoint2 {
    point: Vector2!
    normal: Vector2!
    penetrationDepth: EchoF32!
}

type ContactPatch2 {
    normal: Vector2!
    points: [ContactPoint2!]!
}
```

Open policy question: 2D normals can either live per point or only on the
patch. The first implementation should keep both for shape parity and easier
debugging.

### Overlap result

```graphql
type OverlapResult3 {
    overlaps: Boolean!
    patch: ContactPatch3
}
```

If `overlaps == false`, `patch` must be absent. If `overlaps == true`, `patch`
must be present unless the pair is explicitly marked "boolean-only" by the
query request.

### Raycast result

```graphql
type RayHit3 {
    hit: Boolean!
    time: EchoF32
    point: Vector3
    normal: Vector3
}
```

`time` is distance along the ray direction after direction normalization. The
normal is a world-space surface normal at the hit point. A miss has no time,
point, or normal.

Segment casts use the same result shape but clamp time to `[0, 1]` over the
segment.

### Swept collision result

```graphql
type SweepHit3 {
    hit: Boolean!
    time: EchoF32
    patch: ContactPatch3
}
```

`time` is normalized over the sweep interval: `0` at the start pose, `1` at the
end pose. A hit at `0` represents initial overlap.

## Determinism and canonicalization

Every scalar entering geometry must pass the Echo float policy:

- `-0.0` canonicalizes to `+0.0`.
- All NaN payloads canonicalize to one quiet NaN bit pattern.
- Subnormals canonicalize according to `warp-math` scalar policy.
- Canonical bytes are little-endian fixed-size encodings.

Shape validation runs after scalar canonicalization.

Initial validation rules:

| Shape                   | Validation                                                        |
| ----------------------- | ----------------------------------------------------------------- |
| `Aabb2` / `Aabb3`       | `min <= max` component-wise                                       |
| `Obb3` / `Box2`         | half extents finite and non-negative                              |
| `Sphere3` / `Circle2`   | radius finite and non-negative                                    |
| `Cylinder3`             | axis finite and non-degenerate; radius / half-height non-negative |
| `Plane3` / `HalfSpace*` | normal finite and non-degenerate                                  |
| `Ray*`                  | direction finite and non-degenerate                               |
| `Triangle*`             | non-degenerate area unless explicitly marked degenerate-ok        |
| `Segment*`              | endpoints finite; zero-length segment rejected by default         |
| `Polygon2`              | at least three vertices; first slice can require convex polygons  |
| `ConvexHull3`           | at least four non-coplanar vertices; closed plane set             |

Degenerate policy must be explicit. Silent normalization at raw decode is
forbidden unless the type contract says the field is normalized by definition.

## Algorithm and solver surface

The geometry, collision, and lighting kernel should expose seven families of
deterministic work:

1. Overlap tests.
2. Ray / segment intersection tests.
3. Swept collision tests.
4. Collision solver primitives.
5. Broad-phase machinery.
6. Constraints, joints, and stacking support.
7. Lighting and optics calculations.

### Overlap tests

The first implementation should not attempt every pair in the full Cartesian
product. It should add pairs in a dependency-driven order:

1. `Aabb*` versus `Aabb*`.
2. `Sphere3` / `Circle2` versus AABB.
3. `Sphere3` / `Circle2` versus same-kind.
4. `Ray*` and `Segment*` intersection against simple shapes.
5. Triangle support needed by jedit meshes.
6. OBB / Box SAT.
7. Cylinder, plane, half-space.
8. Convex hull support mapping and clipping.

Every pair implementation must declare:

- supported dimensionality,
- symmetric or ordered normal convention,
- degeneracy behavior,
- exact fixture cases,
- broad-phase compatibility.

### Ray intersection tests

Raycast functions must return the nearest positive hit by default. Batch query
APIs must also support:

- first hit,
- any hit,
- all hits sorted by time and shape id,
- ignored shape id,
- maximum time / distance,
- front-facing-only triangle or plane options.

The jedit title scene currently needs:

- ray versus sphere,
- ray versus cube / AABB / OBB-like box,
- ray versus mesh triangles,
- ray versus floor plane,
- broad-phase object bounds,
- ignored object for reflections,
- front-facing mesh queries for primary title rays.

That is the minimum migration target.

### Swept collision tests

Swept collision tests must return normalized time and contact patch. Initial
sweep support can be conservative:

1. swept AABB using slab intervals,
2. swept sphere/circle using relative raycast,
3. swept segment/triangle only after raycast fixtures are stable,
4. convex sweep once support mapping is in place.

Broad-phase uses fat AABBs to find candidate pairs. Narrow-phase owns the
actual time and contact.

### Collision solvers

Collision solvers turn shape queries into stable contact information and,
later, motion corrections. Echo should expose solver primitives before it
exposes a full rigid-body world.

Initial solver families:

- **SAT:** Separating Axis Theorem for convex 2D boxes, polygons, 3D OBBs, and
  convex hulls where candidate axes are explicit and fixture-proven.
- **GJK:** Gilbert-Johnson-Keerthi distance / overlap for convex shapes through
  deterministic support mappings.
- **EPA:** Expanding Polytope Algorithm for penetration depth after GJK reports
  overlap, added only after GJK simplex ordering is stable.
- **Swept GJK:** conservative advancement over convex support mappings for
  time-of-impact queries.
- **Clipping:** contact patch generation from SAT/EPA features, with stable
  point ordering and documented feature ids.

SAT should land before GJK because it gives readable fixtures for boxes and
polygons. GJK should then generalize convex support mapping for hulls,
cylinders approximated by support policy, and future author-defined convex
shapes.

Determinism requirements:

- Candidate axes sort by stable feature id, not by discovery order.
- Simplex vertices sort by insertion step and support direction tie-breakers.
- Ties choose the smallest canonical shape id / feature id.
- Contact points sort lexicographically after scalar canonicalization.
- Iteration caps are fixed and reported in the solver receipt.
- Failure modes return typed obstructions, never ambient panics.

### Constraints and joints

Constraints and joints are deterministic relationships over bodies,
transforms, velocities, and contact patches. They are not renderer concepts.

Initial constraint surface:

- contact normal constraint,
- friction tangent constraint,
- distance constraint,
- point-to-point constraint,
- orientation alignment constraint,
- limit constraint,
- motor target constraint,
- spring / damper constraint.

Initial joint surface:

- fixed joint,
- revolute / hinge joint,
- prismatic / slider joint,
- ball-and-socket joint,
- distance joint,
- spring joint.

The first implementation does not need to solve all of these. It does need to
reserve the schema vocabulary and define how constraints become causal
evidence:

```text
ConstraintInput
  -> SolverPolicy
    -> ConstraintImpulse / PositionCorrection
      -> SolverReceipt
```

Solver receipts must include:

- policy id,
- iteration count,
- convergence status,
- sorted constraint ids,
- accumulated impulses or corrections,
- residual error summary.

### Stacking and resting contact

Stacking is where deterministic collision becomes visibly useful. It is also
where sloppy solvers become nondeterministic.

Echo should treat stacking as a first-class validation ladder:

1. one box resting on an infinite plane,
2. two AABBs stacked vertically,
3. a small tower of AABBs,
4. rotated boxes with SAT contacts,
5. mixed circle / box or sphere / AABB stacks,
6. sleeping / waking policy only after contact persistence is stable.

Stacking policy requirements:

- persistent contact ids across ticks,
- deterministic manifold reduction when there are too many contact points,
- fixed solver iteration counts unless policy explicitly varies them,
- stable warm-start ordering if warm starting is enabled,
- explicit gravity and timestep in query / simulation input,
- fixture fingerprints for final transforms, velocities, contact patches, and
  solver receipts.

Sleeping is a later feature. The first stacking slices should keep all bodies
awake so contact math is easier to inspect.

## Broad-phase and acceleration structures

Acceleration is an execution artifact. It may be retained as evidence, but it
does not change geometry truth.

Initial acceleration surface:

- per-shape bounding sphere / AABB,
- scene-level AABB / sphere,
- deterministic broad-phase pair ordering,
- ray batch candidate filtering,
- sweep-and-prune,
- dynamic AABB tree,
- static mesh BVH,
- optional spatial hash for large uniform scenes.

Rules:

- Acceleration may reject impossible candidates.
- Acceleration must never reject a true hit or overlap.
- Accelerated and unaccelerated query results must be byte-identical.
- Acceleration cache identity includes shape bytes, transform bytes, schema id,
  and build policy id.
- Broad-phase pair order is canonical and independent of insertion order.
- Dynamic tree rotations, rebuild thresholds, and fat-margin policies are part
  of the acceleration policy id.
- Static BVH construction uses deterministic split rules and stable primitive
  ids.

The jedit title-scene broad-phase cache should become a test input for this
surface. Its current local behavior is useful pressure, not final architecture.

## Lighting and optics calculations

Lighting is adjacent to geometry but should not erase the boundary between
deterministic math and renderer presentation. Echo should own canonical
lighting and optics calculations when those calculations are part of a retained
ray-tracing or simulation witness. Applications still own art direction,
material choice, tone mapping, terminal glyphs, and presentation.

Initial lighting surface:

- ambient light,
- directional light,
- point light,
- spot light,
- emissive material,
- Lambert diffuse term,
- Phong or Blinn-Phong specular term,
- Fresnel term,
- reflection ray,
- refraction ray,
- shadow ray visibility,
- simple fog / extinction term,
- deterministic color-space conversion helpers.

Initial ray-tracing surface:

```text
RayBatch3
  -> GeometryHitBatch3
    -> LightingQuery3
      -> ShadingSample3
```

The first lighting implementation should stay deliberately small:

- direct lighting only,
- one bounce for reflection / refraction only when requested,
- no stochastic sampling,
- no path-tracing integrator,
- no ambient occlusion unless it is deterministic and fixture-proven.

GPU acceleration is allowed only as an acceleration backend. The canonical
authority remains the deterministic CPU/reference implementation until a GPU
backend can prove byte-equivalent output or an explicitly documented tolerance
policy. GPU backend receipts must name:

- backend id,
- shader/kernel digest,
- input schema id,
- output schema id,
- precision policy,
- equivalence fixture set.

For jedit, Echo lighting should initially cover the math currently embedded in
title-scene optics: direct light direction, surface normals, reflection,
refraction, shadow visibility, fog, and simple color mixing. jedit should still
decide the theme colors, material palette, terminal dithering, and final
surface composition.

## Jedit migration plan

jedit should stop owning geometry algorithms in title-scene rendering. It
should own authored scene data and presentation, then call Echo geometry.

Current jedit-local areas to migrate:

- `src/ui/title-scene-hit.ts`
- `src/ui/title-scene-mesh-hit.ts`
- `src/ui/title-scene-cube.ts`
- `src/ui/title-scene-ray-acceleration.ts`
- ray setup and sampling code in `src/ui/title-screen.ts`
- reflected ray usage in `src/ui/title-screen-optics.ts`
- floor shadow ray usage in `src/ui/title-floor-light-effects.ts`

Target split:

| Concern                                    | Owner         |
| ------------------------------------------ | ------------- |
| title mesh asset loading                   | jedit         |
| title scene object list                    | jedit         |
| material palette / art direction           | jedit         |
| terminal sampling / braille / ASCII        | jedit         |
| shape conversion to Echo geometry payloads | jedit adapter |
| ray / shape intersection                   | Echo          |
| broad-phase acceleration                   | Echo          |
| reusable lighting / optics math            | Echo          |
| batch ray query result semantics           | Echo          |
| cross-language fixture proof               | Echo          |

The first production cut should use a batch query API, not one WASM call per
ray. Per-ray FFI would make the boundary correct but slow.

Proposed query shape:

```graphql
type GeometryScene3 {
    shapes: [ShapeInstance3!]!
}

type ShapeInstance3 {
    id: GeometryShapeId!
    shape: Shape3!
    transform: Transform3D
}

type RayBatch3 {
    rays: [Ray3!]!
    mode: RayQueryMode!
    maxTime: EchoF32
}

enum RayQueryMode {
    ANY_HIT
    FIRST_HIT
    ALL_HITS
}

type RayBatchHit3 {
    rayIndex: Int!
    shapeId: GeometryShapeId
    hit: RayHit3!
}
```

The batch result must preserve input ray order. Ties within a ray sort by
canonical shape id.

## Causality posture

Geometry values are not just helper structs. In Echo they can be causal
evidence.

Examples:

- A `RayBatch3` query can be retained as a reading artifact with schema id,
  query bytes, acceleration policy id, and result bytes.
- A swept collision can produce a deterministic contact patch that explains a
  later motion decision.
- A jedit render witness can prove that visible title-scene pixels were
  derived from a particular geometry scene and ray batch policy.

Rules:

- Geometry query requests are canonical byte payloads.
- Geometry query responses are canonical byte payloads.
- Acceleration structures are derived from request state; they are not
  independent truth unless retained with their derivation basis.
- If two branches build different acceleration structures from identical
  shape bytes and policy ids, that is a deterministic bug.

## Acceptance criteria

- [ ] Echo has authored SDL for linear algebra, 2D shapes, 3D shapes, and
      geometry query results.
- [ ] Wesley emits Rust and TypeScript surfaces for the initial math and shape
      set.
- [ ] Generated codecs use canonical little-endian bytes with stable numeric
      discriminants.
- [ ] `warp-math` remains the scalar and linear algebra authority.
- [ ] `warp-geom` becomes the shape and query authority.
- [ ] Accelerated and unaccelerated ray query fixtures produce identical
      canonical results.
- [ ] Contact patches define normal direction, point coordinates, and
      penetration-depth semantics.
- [ ] Swept collision results define normalized time and initial-overlap
      behavior.
- [ ] SAT fixtures define axes, feature ids, contact patch ordering, and normal
      orientation.
- [ ] GJK and swept GJK fixtures define simplex tie-breakers, iteration caps,

- [ ] Constraint and joint schemas exist with deterministic solver receipts.
- [ ] Stacking fixtures prove stable resting contact under fixed policy ids.

## Risks / unknowns

- **Pair explosion:** Full all-shapes overlap/raycast/sweep support is a large
  matrix. The cycle must land pair subsets deliberately.
- **TypeScript performance:** jedit needs batch queries. Per-ray FFI is not
  acceptable for title rendering.
- **NaN and subnormal policy:** JavaScript cannot preserve arbitrary NaN
  payloads. Echo canonicalization must make that irrelevant at the boundary.
- **Convex hull validation:** Robust hull validation is non-trivial. The first
  convex slice can require precomputed planes and strict fixtures.
- **Triangle mesh ownership:** Echo should own triangle query semantics but not
  application asset loading. jedit can load meshes and submit triangles or
  retained mesh artifacts.
- **Contact manifold determinism:** Contact clipping can be order-sensitive.
  Point order, tie-breakers, and normal orientation must be fixture-proven.
- **GJK non-convergence:** GJK and swept GJK need fixed iteration caps, stable
  simplex tie-breakers, and typed obstruction output.
- **Constraint solver drift:** Sequential impulse / projected Gauss-Seidel
  style solvers can diverge if ordering, iteration count, or warm-start data is
  implicit.
- **Stacking sensitivity:** Resting stacks magnify tiny normal and penetration
  differences. Stacking must be a fixture ladder, not an anecdotal visual test.
- **Broad-phase false negatives:** Acceleration bugs can erase real contacts.
  Every broad-phase policy needs accelerated/unaccelerated equivalence tests.
- **GPU determinism:** GPU acceleration can be useful but must not silently
  become geometry or lighting authority. Backend precision policy and fixture
  equivalence must be explicit.
- **Lighting scope creep:** A deterministic direct-light reference is useful.
  A full physically based renderer or path tracer is a separate product-sized
  project.
- **Existing `echo-scene-port`:** The scene port is render-oriented and should
  not become the geometry kernel by accident. It may adapt to generated
  geometry types later.

## Non-goals

- Shipping a production-grade general game physics engine in the first
  implementation slice.
- Moving jedit lighting, materials, terminal rendering, or product UX into
  Echo.
- Making Three.js or another renderer an Echo dependency.
- Treating JSON as canonical geometry payload.
- Generating every possible shape-pair algorithm before jedit's first Echo
  geometry migration.
- Allowing acceleration structures to change query meaning.
- Letting constraints or joints mutate bodies without retained solver receipts.
- Making GPU output canonical before a CPU/reference equivalence policy exists.
- Building a full stochastic path tracer as part of the first lighting slice.

## Open questions

- Should the generated contract use `Vector3` or shorter `Vec3` names?
  Current recommendation: `Vector3` at the GraphQL boundary, `Vec3` in Rust.
- Should 2D geometry live in the same crate as 3D geometry or in dimensional
  modules under `warp-geom`?
- Should `ContactPatch` store per-point normals, one patch normal, or both?
  Current recommendation: both, for debug clarity and pair-specific evidence.
- Should `Plane3.offset` use `dot(normal, point) = offset` or
  `dot(normal, point) + offset = 0`? Pick one before schema lands.
- How should retained mesh artifacts reference large vertex buffers without
  duplicating bytes in every ray batch?
- Which jedit title-scene fixture becomes the first Echo geometry migration
  witness?
- Should the first constraint solver use sequential impulses, XPBD-style
  position constraints, or a narrower contact-only solver?
- What is the minimum deterministic warm-start story for stacking, and should
  warm starting be disabled until contact persistence is proven?
- Which broad-phase policy should become the first non-reference production
  policy: sweep-and-prune, dynamic AABB tree, or static BVH for meshes?
- Which lighting model should be the first Echo reference: Lambert plus
  Blinn-Phong, or a smaller Lambert-only direct-light model?
- What tolerance, if any, is acceptable for GPU lighting equivalence, and when
  must the result be byte-identical?
